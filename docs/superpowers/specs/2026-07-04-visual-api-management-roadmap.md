# 可视化接口管理（Apifox 式）— 路线图与范围分解

> 日期：2026-07-04 | 状态：规划中（各子项目另立 spec） | 蓝本：Apifox 的接口管理体验

## 1. 愿景与定位

在现有「GitHub 仓库为存储、Monaco + swagger-ui 双栏编辑」的基础上，为 OpenAPI 文档
提供 Apifox 式的**结构化可视界面**：接口按业务分组浏览、参数与响应以表格/表单呈现、
数据模型可视化管理、（远期）表单化编辑与在线调试。

与现状的关系：**不替换**现有源码编辑页，而是新增「可视模式」。源码模式（Monaco）
仍是权威编辑入口与兜底；可视模式从同一份 YAML/JSON 解析而来，两者共享 fileQuery
数据链路。存储与版本模型不变（仍是 Git 提交）。

## 2. 现有地基（已具备）

- 文档获取/保存链路：`fileQuery`（含 SHA 乐观锁、409 处理）、保存 mutation。
- 解析层：`lib/openapi.ts` 的 `parseDocument`（js-yaml，YAML/JSON 统一）。
- 校验层：Spectral oas 规则集实时诊断（可在可视模式复用，如接口卡片上标违规数）。
- 版本层：提交历史抽屉 + diff + 回滚。
- UI 体系：shadcn/ui + Tailwind，编辑页已有 header/双栏/抽屉的布局骨架。

## 3. 子系统分解（每块独立 spec → 计划 → 实现）

### 3.1 接口浏览（只读可视化）——建议第一期

- **范围**：编辑页新增「可视/源码」模式切换。可视模式：左侧接口树（按 `tags` 分组，
  无 tag 归「未分组」；节点 = method 徽标 + path + summary），右侧接口详情——
  路径/描述、parameters 表格（名称/位置/类型/必填/说明）、requestBody 与各状态码
  response 的 schema 结构化展开（含 `$ref` 解析、嵌套对象/数组、枚举、示例）。
- **核心资产**：`lib/openapi-ir.ts`——把解析后的文档规整为稳定的中间结构
  （OperationSummary / ResolvedSchema 等），**后续三块全部复用**；`$ref` 本地解析
  （`#/components/...`，跨文件 ref 一期不做）。
- **依赖**：无新服务端；可能新增少量 UI 组件（tabs、collapsible、table）。
- **主要风险**：OpenAPI 2/3.0/3.1 差异（一期以 3.x 为主，2.0 降级提示）；循环 `$ref`
  的展开截断策略。

### 3.2 数据模型管理

- **范围**：可视模式加「数据模型」区：`components/schemas` 列表、单个 schema 的
  字段树、被哪些接口引用（反向引用索引）。
- **依赖**：3.1 的 IR 与 schema 渲染组件直接复用。
- **主要风险**：低——只读呈现为主。

### 3.3 可视化编辑（表单 → YAML 回写）

- **范围**：在 3.1/3.2 的详情页上开放编辑：改 summary/描述/参数行、增删参数、
  编辑 schema 字段，改动**回写到 YAML 源文本**后走现有保存流（dirty/SaveDialog/409
  全复用）。
- **关键技术决策（届时 spec 敲定）**：回写策略——AST 级精准修改（保留注释与格式，
  需 yaml 库的 CST 能力，复杂）vs 结构序列化重排（简单但会破坏手写格式/注释）。
  这是本块最大的坑，必须先做技术验证（spike）再定 spec。
- **依赖**：3.1 的 IR 需要携带「源码位置」信息（node → YAML 路径映射）。
- **主要风险**：高——双向同步的一致性；可视编辑期间用户切到源码模式改文本的合流。

### 3.4 接口调试（在线发请求）

- **范围**：接口详情页「试一试」：按参数表单填值、选 server、发请求、看响应
  （状态码/耗时/响应体高亮）。
- **硬约束（必须先接受再立项）**：纯前端站点受浏览器 CORS 限制——只能调
  **显式允许跨域**的目标 API；不允许跨域的服务会被浏览器直接拦截，且无法绕过。
  Apifox 靠桌面客户端/本地代理解决，本项目无服务端。可选缓解：
  a) 仅支持 CORS 友好的 API 并明确提示；b) 用户自建轻量代理（提供 docker/worker
  模板，与 OAuth Exchange 函数可共用基建）；c) 做浏览器扩展（超出当前定位）。
- **依赖**：3.1 的参数表单结构；若走代理方案则与 OAuth Exchange 子项目共享部署基建。
- **主要风险**：CORS 体验落差导致「看起来坏了」；鉴权头/敏感信息在浏览器侧的处理。

## 4. 建议顺序与理由

**3.1 → 3.2 → 3.3 → 3.4**

- 3.1 建立 IR 解析层与可视界面骨架，是其余三块的公共地基，且独立可交付
  （纯只读，无一致性风险）。
- 3.2 几乎是 3.1 组件的低风险复用，顺势收割。
- 3.3 价值最大但坑最深（YAML 回写保格式），放在界面与 IR 稳定之后，先 spike 后立项。
- 3.4 受 CORS 硬约束，是否做、做到什么程度，等 a/b 方案与用户预期对齐后再定。

## 5. 全局技术约束（各期 spec 继承）

- 纯前端不变：所有能力在浏览器内完成，不引入必须的服务端（3.4 的代理为可选项）。
- 大依赖一律动态 `import()` 独立 chunk（沿用 Spectral 先例）；编辑页主 chunk 已 1.35MB，
  可视模式组件应懒加载。
- `lib/` 纯函数无 React 依赖可单测；`features/` 消费 lib；`routes/` 只装配——分层不破。
- 源码模式始终可用：可视模式解析失败（YAML 语法错/非 OpenAPI 文档）时降级提示
  并引导回源码模式，不得让用户被锁在坏状态里。
- 中文文案；vp 工具链；routeTree.gen.ts 勿手改。

## 6. 开放问题（进入各期 brainstorm 时逐个敲定）

1. 3.1 的模式切换粒度：编辑页内 Tab 切换，还是独立路由（`/view/*` 与 `/edit/*`）？
2. 3.1 左侧接口树与现有「文件树」的关系：同一侧栏两段式，还是进入文件后整页切换？
3. 3.3 的回写技术选型：yaml CST（如 `yaml` 包的 document API）vs 重序列化，需 spike。
4. 3.4 是否值得做浏览器内直调（仅 CORS 友好 API），还是直接等代理基建。
5. OpenAPI 2.0 文档的支持深度（只提示升级，还是做兼容映射）。
